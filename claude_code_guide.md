## 1. 什么是 Claude Code

Claude Code 是 Anthropic 官方推出的 **CLI 编程助手**，它不只是一个聊天窗口，而是能在你的终端或 VSCode 里**直接读写文件、执行命令、操作 Git** 的 Agentic 工具。

### 与普通 AI 对话框的区别

| 对比维度 | 普通 AI 对话框 | Claude Code |
|---|---|---|
| 操作方式 | 你复制代码，手动修改 | Claude 直接读写你的文件 |
| 上下文获取 | 你手动粘贴代码 | Claude 自主搜索、阅读相关文件 |
| 执行能力 | 只给建议 | 可执行 Shell 命令、运行测试 |
| Git 支持 | 无 | 可直接 commit、创建 PR |

### 工作方式

Claude Code 以循环方式自主完成任务：

```
读取相关文件 → 修改代码 → 运行验证 → 重复直到完成
```

你可以在任意阶段按 `Ctrl+C` 打断并重新指导 Claude。

## 2. 安装与启动

### 安装

```bash
npm install -g @anthropic-ai/claude-code
```

### 登录

```bash
claude auth login   # 使用 Claude Pro 账号登录
```

### 启动方式

```bash
claude          # 在当前目录启动新会话
claude -c       # 继续上一次会话（最常用）
```

## 3. 核心功能介绍

### 文件操作

Claude Code 可以直接操作你本地的文件，无需手动粘贴代码：

- 自主读取项目中的相关文件
- 精准修改代码，仅改动必要部分
- 创建新的源文件、测试文件

### 命令执行

```
编译运行      g++ -std=c++17 -o output main.cpp
运行测试      npm test / pytest
Git 操作      自动完成 add、commit、push
```

> ⚠️ 执行涉及删除或推送等危险操作前，Claude 默认会先询问你确认。

### 代码搜索与分析

```
找到某函数的定义在哪
分析这个模块的调用关系
解释这段代码的逻辑
帮我找出所有处理 xxx 的地方
```

## 4. 高效撰写代码的方法

Prompt 的质量直接决定输出质量，这是本文最重要的一节。

### 原则一：具体胜于模糊

| 质量 | 示例 |
|---|---|
| ❌ 差 | `fix the bug` |
| ❌ 差 | `优化一下这段代码` |
| ✅ 好 | `用户反馈登录后跳转失败，复现步骤是 session 超时后点击登录按钮。请检查 src/auth/ 下的 token 刷新逻辑，先写一个能复现 bug 的测试用例，再修复它。` |

### 原则二：提供验收标准

告诉 Claude 什么叫"完成"：

```
实现一个 HTTP 客户端封装类，要求：
1. 支持 GET / POST 方法
2. 超时时间默认 5s，可配置
3. 返回结果用自定义的 Response 结构体包装
4. 所有功能需要通过 test/http_client_test.cpp 中的测试用例
```

### 原则三：引用具体文件

```
✅ 请检查 modules/function/narrow_scene_processor.cpp 中
   checkODTrigger 函数的对障检测逻辑是否正确

❌ 检查一下障碍物检测那个函数
```

### 推荐的四阶段工作流

对于较复杂的功能，建议按以下步骤推进：

**第一步：探索（只读）**

按 `Shift+Tab` 切换到 **Plan Mode**，让 Claude 先读代码，不做任何修改：

```
请阅读 modules/function/ 目录，理解窄路场景检测的整体流程。
完成后告诉我：各函数的调用关系，以及你发现的潜在问题。
```

**第二步：规划（讨论方案）**

```
基于你的分析，我需要新增倒车时的后方障碍检测功能。
请给出实现方案，说明需要改哪些函数，不要动代码。
```

**第三步：实现（正式编码）**

确认方案后，切换回默认模式：

```
方案确认，请开始实现。每修改一个函数后说明改了什么。
实现完成后编译确认没有语法错误。
```

**第四步：收尾（提交）**

```
运行相关测试，确认没有回归。
用中文写清晰的 commit message，提交代码。
```

### 迭代反馈技巧

```
# ❌ 重新提问会浪费上下文
"这不对，重来"

# ✅ 精准纠正，保留上下文
"你修改的 doProcess 函数有问题：速度防抖部分，
 nr_over_spd 的清零位置放错了，应该在 else 分支里，
 请只修正这一个问题，其他不要动。"
```

## 5. CLAUDE.md — 项目配置文件

`CLAUDE.md` 是每次会话自动加载的**项目配置文件**，用来告诉 Claude 这个项目的规范和约定，是团队共享规范的最佳载体。

### 文件位置

```
./CLAUDE.md              项目级（推荐提交到 git，团队共享）
~/.claude/CLAUDE.md      用户级（所有项目生效，个人偏好）
```

### 应该写什么

```markdown
# CLAUDE.md 示例

## 编译命令
\```bash
g++ -std=c++17 -o output main.cpp   # 编译
./run_tests.sh                       # 运行测试
\```

## 代码规范
- 所有注释使用中文
- 类名 PascalCase，函数名 camelCase，变量名 snake_case
- 禁止使用裸指针，优先使用智能指针

## 项目结构
- modules/function/    HMI 功能处理逻辑
- modules/perception/  感知数据处理

## 重要约定
- 修改 .proto 文件后需重新运行 protoc
- 坐标系使用车体坐标系（x 轴朝前，y 轴朝左）
```

### 初始化命令

让 Claude 自动分析项目并生成初始 CLAUDE.md：

```
/init
```

> 建议保持 CLAUDE.md 在 200 行以内，内容越精简，Claude 读取效率越高。

## 6. Git 工作流

Claude Code 内置 Git 操作支持，并有安全保护机制。

### 基本操作示例

```
查看改动    "现在有哪些文件改动？"
提交代码    "提交这次修改，commit message 用中文写清楚改了什么"
创建 PR     "把当前分支推上去，创建一个 PR，描述用中文写"
```

### Claude 的 Git 安全原则

Claude Code 内置了以下保护规则：

| 操作 | Claude 的行为 |
|---|---|
| `git push --force` | 会警告风险，询问是否确认 |
| `git reset --hard` | 会提示这是破坏性操作 |
| 提交含密钥的文件 | 主动警告，拒绝提交 |
| 跳过 pre-commit 钩子 | 不会主动跳过，提示先修复根本原因 |

### 标准提交流程

```
1. 实现完成后：
   "运行编译，确认没有报错"

2. 确认无误后：
   "用中文写 commit message 并提交，格式：
    [功能/修复/重构] 简短标题

    - 改了什么
    - 为什么这样改"
```

## 7. 权限模式

按 `Shift+Tab` 可以在会话中循环切换权限模式：

| 模式 | 说明 | 适用场景 |
|---|---|---|
| **Default（默认）** | 每次操作前都询问你 | 日常使用，推荐 |
| **Accept Edits** | 文件编辑自动接受，命令仍询问 | 信任代码修改内容时 |
| **Plan Mode** | 只读，不修改文件不执行命令 | 分析代码、讨论方案 |

> Plan Mode 是探索陌生代码库的利器，进入后 Claude 只能阅读，不会误改任何文件。

## 8. 上下文管理

理解上下文是高效使用 Claude Code 的关键。

### 什么是上下文

Claude 的"记忆"包含以下所有内容：

```
CLAUDE.md 内容 + 对话历史 + Claude 读过的文件 + 命令输出结果
```

Pro 套餐的上下文窗口约为 **200K tokens**（约 15 万汉字）。当上下文过长时，Claude 会自动压缩早期内容，可能导致之前说过的细节被遗忘——这就是为什么重要规范要写进 CLAUDE.md 而不是只在对话里说一次。

### 上下文管理命令

```
/context    查看当前上下文使用量
/clear      清空上下文，重新开始（换任务时使用）
/compact    压缩历史对话，释放空间（不丢失关键信息）
```

> **建议**：完成一个独立任务后，立即用 `/clear` 重置上下文，再开始下一个任务。上下文越干净，Claude 越聚焦。

### 会话恢复

```bash
claude -c       # 继续最近一次会话
/rename 任务名  # 给当前会话命名，方便后续找到
```

|---|---|

## 11. 在 VSCode 中使用

Claude Code 提供 VSCode 插件，可以不离开编辑器直接使用，并具备图形化差异对比等专属功能。

### 打开 Claude Code 面板

| 方式 | 操作 |
|---|---|
| 编辑器工具栏 | 点击右上角 **✦ 火花图标**（需有文件打开）|
| 活动栏 | 点击左侧边栏的 **✦ 图标** |
| 命令面板 | `Ctrl+Shift+P` → 搜索 `Claude Code` |

> 💡 建议将 Claude 面板拖到 **右侧次级边栏**，代码和 Claude 同屏显示效率最高。

### VSCode 专属功能

#### @-mention 引用文件

在输入框中直接引用文件，无需手动复制粘贴：

```
@narrow_scene_processor.cpp      引用整个文件
@modules/function/               引用整个目录
@main.cpp#10-50                  只引用第 10 到 50 行
```

快捷插入当前光标所在文件的引用：**`Alt+K`**（Mac: `Option+K`）

#### 图形化差异对比

Claude 修改文件时，以**并排对比**展示改动，可逐块接受或拒绝，比终端版更直观。

#### 权限模式切换

点击输入框底部的模式指示器即可切换，与终端版 `Shift+Tab` 效果相同。

### VSCode 快捷键

| 快捷键 | 功能 |
|---|---|
| `Ctrl+Esc` | 编辑器与 Claude 面板之间切换焦点 |
| `Alt+K` | 插入当前文件的 @-mention 引用 |
| `Ctrl+Shift+Esc` | 新开一个 Claude 标签页 |
| `Shift+Enter` | 输入框内换行，不提交 |

### VSCode 插件 vs 终端 CLI

| 功能 | VSCode 插件 | 终端 CLI |
|---|---|---|
| 差异对比 | ✅ 图形化并排对比 | 终端文本 |
| 文件引用 | ✅ @-mention + 拖拽 | @-mention |
| 多会话 | ✅ 多标签页 | 多开终端 |
| 完整命令支持 | 部分 | 全部 |

**两者可以结合使用：** 在 VSCode 内置终端（`` Ctrl+` ``）中运行 `claude`，同时享受终端完整命令 + IDE 图形化差异对比。

## 12. 常见误区与最佳实践

### 常见误区

#### ❌ 一次性布置超大任务

```
# 错误
"帮我重构整个项目，把所有全局变量改成类成员，
 同时把注释改成英文，还要给每个函数加单元测试"

# 正确：逐步推进
"先帮我找出 modules/function/ 里所有的全局变量，列个清单"
（确认后）"先把 narrow_scene_processor.cpp 里的全局变量改成静态类成员"
```

#### ❌ 换任务不重置上下文

```
# 错误：在分析 A 模块的会话里突然问 B 模块
（分析完窄路检测后）"帮我看看交通灯处理的逻辑"

# 正确：先清空
/clear
"现在帮我分析 traffic_light2d_processor.cpp 的逻辑"
```

#### ❌ 无条件信任 Claude 的修改

Claude 修改文件后，养成 `git diff` 检查的习惯，确认改动符合预期再提交。

#### ❌ 把规范只说在对话里

上下文会被压缩，之前说过的规范可能被遗忘。**应该写入 CLAUDE.md** 持久保存。

### 最佳实践清单

```
✅ 在 CLAUDE.md 中写明：编译命令、代码规范、项目结构
✅ 完成任务后 /clear，保持上下文干净
✅ 复杂任务先在 Plan Mode 分析，再切回来执行
✅ 给 Claude 提供具体的验收标准
✅ 修改后用 git diff 检查，确认无误再提交
✅ 用 /rename 给会话命名，方便后续 claude -c 恢复
```
